# 性能测试指引

本文用于说明本项目实验报告中性能测试部分的设计方法，重点对应以下两类图表：

1. 不同加密场景下的端到端延迟对比
2. 不同传感器数量下的吞吐量对比

## 1. 测试目标

性能测试需要回答两个问题：

1. 引入 TLS 与应用层 AES-GCM 后，端到端时延增加了多少。
2. 当传感器数量增加时，系统整体吞吐量如何变化。

因此建议将性能测试拆成两组：

- 延迟测试：关注单条消息从发送到被中心处理完成的耗时
- 吞吐量测试：关注单位时间内中心能够处理多少条有效消息

## 2. 与当前仓库实现的对应关系

当前仓库已经具备以下测试基础：

- Broker 默认运行在 `8883` 端口，并启用 TLS/mTLS，见 [config/mosquitto.conf](/Users/kannazuki/Dev/mine-sensor-secure-comm/config/mosquitto.conf:1)
- 传感器支持 `--plaintext` 模式，可用于“仅 TLS”基线测试，见 [src/mine_sensor_secure_comm/sensor_cli.py](/Users/kannazuki/Dev/mine-sensor-secure-comm/src/mine_sensor_secure_comm/sensor_cli.py:27)
- 默认模式下，传感器会对业务 payload 做 AES-GCM 加密，再通过 TLS 发送
- 消息中已写入 `send_time_ns`，可用于端到端延迟计算，见 [src/mine_sensor_secure_comm/sensor_cli.py](/Users/kannazuki/Dev/mine-sensor-secure-comm/src/mine_sensor_secure_comm/sensor_cli.py:81) 和 [src/mine_sensor_secure_comm/sensor_cli.py](/Users/kannazuki/Dev/mine-sensor-secure-comm/src/mine_sensor_secure_comm/sensor_cli.py:87)
- 仓库文档中已经定义了延迟统计思路，见 [doc/system_design.md](/Users/kannazuki/Dev/mine-sensor-secure-comm/doc/system_design.md:347)

需要注意的是：

- 当前仓库默认没有“无 TLS、无 AES-GCM”的现成运行路径
- 当前中心端没有直接输出完整的端到端延迟统计结果

因此，如果报告一定要画“无加密 / 仅 TLS / TLS+AES-GCM”三组柱状图，就需要额外补一个无 TLS 的测试场景，以及一个用于统计 `receive_time_ns - send_time_ns` 的采集逻辑。

## 3. 图 1：不同加密场景下的端到端延迟对比

### 3.1 测试目的

该图用于体现安全机制引入后对系统实时性的影响。

### 3.2 建议测试场景

建议设置以下三组场景：

1. 无加密
2. 仅 TLS
3. TLS + AES-GCM

三组场景的含义如下。

#### 场景 A：无加密

- MQTT 走普通 TCP
- 不启用 TLS
- 不启用应用层 AES-GCM
- 作为最基础的性能基线

该场景不属于当前仓库默认配置，需要额外准备一个明文 MQTT Broker 配置，例如使用 `1883` 端口。

#### 场景 B：仅 TLS

- 保留当前 TLS/mTLS 链路
- MQTT payload 使用明文 JSON
- 传感器使用 `--plaintext`

该场景可直接利用现有代码完成。

#### 场景 C：TLS + AES-GCM

- 保留当前 TLS/mTLS 链路
- MQTT payload 先做 AES-128-GCM 加密
- 再通过 TLS 通道发送

这是当前仓库默认的安全通信模式。

### 3.3 控制变量

为了保证对比有效，三组场景中以下条件应保持一致：

- 传感器数量固定为 `1`
- 消息条数固定，例如 `1000`
- QoS 固定，例如 `QoS 1`
- 单条消息大小尽量一致
- 发送间隔固定
- 测试机器与部署方式固定
- Broker 与地面中心配置除加密方式外尽量一致

### 3.4 延迟定义

单条端到端延迟定义为：

$$
latency = receive\_time\_ns - send\_time\_ns
$$

其中：

- `send_time_ns` 由传感器发送前写入 payload
- `receive_time_ns` 由地面中心在收到并完成处理时记录

### 3.5 建议统计指标

建议至少输出以下指标：

- `count`
- `mean_ms`
- `p50_ms`
- `p95_ms`
- `p99_ms`
- `min_ms`
- `max_ms`
- `drop_count`

柱状图的 Y 轴建议使用：

- 平均延迟 `mean_ms`

正文或表格中再补充：

- `p95_ms`
- `p99_ms`

这样既能体现整体水平，也能体现尾延迟。

### 3.6 推荐实验步骤

1. 启动 Broker 和地面中心
2. 设置单个传感器运行参数
3. 在指定场景下发送固定数量消息，例如 `1000` 条
4. 中心端对每条消息记录 `receive_time_ns`
5. 计算每条消息的端到端延迟
6. 输出统计结果
7. 分别重复场景 A、B、C
8. 将三组 `mean_ms` 画成柱状图

## 4. 图 2：不同传感器数量下的吞吐量对比

### 4.1 测试目的

该图用于体现系统在节点规模增加时的承载能力变化。

### 4.2 建议测试场景

建议固定加密方案为：

- `TLS + AES-GCM`

然后改变传感器数量：

- `1`
- `2`
- `4`

如果报告篇幅充足，也可以对“仅 TLS”再补一张扩展图，但正文主图通常保留一张即可。

### 4.3 为什么不建议直接用默认发送频率

如果吞吐量测试直接沿用自然上报周期，例如 `interval_seconds = 0.5`，那么结果更多反映的是“应用主动发得慢”，而不是“系统处理能力上限”。

因此建议：

- 将发送间隔调小
- 或使用尽可能连续发送的压测方式

这样得到的吞吐量才更接近系统承载能力。

### 4.4 吞吐量定义

设：

- 总有效消息数为 `N`
- 测试总耗时为 `T`

则吞吐量定义为：

$$
throughput = \frac{N}{T}
$$

单位为：

- 消息/秒

这里的 `N` 应只统计中心成功处理的有效消息。

### 4.5 控制变量

为了保证不同传感器数量之间可比，以下条件建议保持一致：

- 加密场景固定为 `TLS + AES-GCM`
- 每个传感器发送相同数量的消息，例如 `1000`
- QoS 固定
- 消息大小固定
- 机器与部署方式固定
- 测试持续时间统计方式一致

### 4.6 推荐实验步骤

1. 固定加密场景为 `TLS + AES-GCM`
2. 先运行 `1` 个传感器，每个发送 `1000` 条消息
3. 记录中心端接收完成所有有效消息的总耗时
4. 计算吞吐量
5. 再分别运行 `2` 个、`4` 个传感器
6. 用三组吞吐量结果绘制柱状图

### 4.7 建议记录的附加指标

除了吞吐量本身，建议同时记录：

- `accepted_count`
- `drop_count`
- `decrypt_error_count`
- `replay_error_count`

这样可以避免“吞吐量看起来很高，但丢包也很多”的误判。

## 6. 建议的结果解读方式

### 6.1 图 1 的解读重点

图 1 主要回答：

- 增加 TLS 是否明显提高了时延
- 在 TLS 基础上再加入 AES-GCM，额外开销有多大

一般可以这样写：

- 无加密延迟最低，作为基线
- 仅 TLS 会带来一定握手与传输保护开销
- TLS + AES-GCM 在仅 TLS 的基础上进一步增加少量处理开销
- 如果增幅可控，则说明系统在提升安全性的同时仍具备较好的实时性

### 6.2 图 2 的解读重点

图 2 主要回答：

- 传感器数量增加后，系统吞吐量如何变化
- 系统是否具有一定的并发扩展能力

一般可以这样写：

- 当传感器数量从 `1` 增加到 `2`、`4` 时，系统总吞吐量会变化
- 如果吞吐量仍保持稳定增长或仅轻微下降，说明系统具有一定扩展性
- 如果下降明显，则说明 Broker、中心处理逻辑或加密验证成为瓶颈

## 8. 实验设计建议总结

本项目性能测试建议遵循以下思路：

1. 用“无加密 / 仅 TLS / TLS+AES-GCM”比较安全机制带来的延迟变化
2. 用“1 / 2 / 4 个传感器”比较节点规模变化带来的吞吐量变化
3. 固定除测试变量外的其他条件，保证结果可比
4. 不仅看平均值，也记录 P95、P99 和丢弃数量
5. 先形成原始测试表，再据此绘制柱状图

按这个结构设计后，图表、表格和文字分析会比较一致，也更适合写入实验报告。
